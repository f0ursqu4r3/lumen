# tragit — Design System

## Design Context

### Users
A solo developer (the owner) using tragit as a daily-driver client over a self-hosted
GitLab instance, triaging issues for defense / space-domain-awareness (SDA) projects.
Usage is focused, repetitive backlog work in an IDE-adjacent context — long sessions,
often a dark editor environment.

### Brand Personality
Precise · calm · operational. A quiet instrument, not a consumer product. It should
feel like trustworthy mission-control software: dense, scannable, fast, with restraint.

### Aesthetic Direction
**Refined dark, Linear-class.** Near-black cool-slate surfaces with subtle elevation;
color carries *meaning* (labels, status, priority) rather than decoration. One rare
amber signal accent. Compact information density. Not the cyan/purple-on-dark AI look.

- **Theme:** dark (chosen for the focused, editor-adjacent viewing context).
- **Density:** compact — maximize scannable rows for backlog triage.

### Type
- **UI / body:** Hanken Grotesk Variable (`--font-sans`) — warm humanist grotesque.
- **Numerals / ids / labels:** Geist Mono Variable (`--font-mono`) — functional tabular
  alignment for issue numbers and counts (not decorative "tech vibes").
- Self-hosted via `@fontsource-variable/*` so they work offline / behind the internal CA.

### Color (defined in `src/styles.css` under `.dark`)
- OKLCH throughout; neutrals tinted toward **hue 256** (cool slate) for cohesion.
- Background `oklch(0.178 0.008 256)`, card `0.214`, border `0.288`. No pure black/white.
- **Accent:** amber `oklch(0.82 0.142 81)` — used sparingly: the primary/create action,
  focus rings, and the header signal dot. ~10% visual weight at most.
- Semantic label/status/priority colors are brightened for contrast on dark surfaces.

### Design Principles
1. **Color means something.** Reserve hue for state (priority/type/status/labels) and the
   single amber accent. Surfaces stay neutral slate.
2. **Lift signal out of noise.** Parse scoped GitLab labels (`scope::value`); promote
   priority/type/status into glyphs and chips, leave the rest as two-tone pills.
3. **Scannable density.** Tight, consistent rows; mono tabular ids; readable contrast
   (luminance-derived text on every colored chip).
4. **No AI tells.** No side-stripe accents (>1px) on list items, no gradient text, no
   glassmorphism. Indicators are leading glyphs / tints / full borders instead.
5. **Quiet motion.** One staggered list reveal on load (ease-out, opacity+transform,
   reduced-motion safe). No perpetual or bouncy animation.

## Reusable pieces
- `src/lib/labels.ts` — scoped-label parsing, `readableText` (WCAG luminance), `darken`,
  `tint`, and `priorityOf` / `typeOf` / `statusOf` / `remainingLabels` semantics.
- `src/lib/issueView.ts` — client-side `sortIssues` (priority/title/updated) and
  `groupIssues` (none/status/priority/assignee), plus the `Facet` type. Sort/group/board
  run on the loaded set because priority & status live in labels (server can't order them).
- `LabelChip.vue` — two-tone scoped pill, contrast-correct, hairline ring.
- `StateBadge.vue` — `compact` dot variant (lists) + full pill (detail).
- `IssueRow.vue` / `IssueCard.vue` — list row & board card. Both use the stretched-link
  pattern (RouterLink overlay + facet buttons above) so clicking a label/priority/status/
  assignee emits a `filter` facet instead of navigating; the row/card still links.

## Issues workspace (not GitLab-shaped)
- **List ⇄ Board** toggle. Board = kanban columns grouped by workflow status
  (on-deck → in-progress → in-review). List = optional grouping + sort menus.
- **Click-to-filter**: any facet on a row/card toggles a filter; active filters show as
  removable tokens. Every facet reduces to a label or assignee filter, so the GraphQL
  query needs no new variables.
- Issues infinite-load (50/page) so client-side sort/group sees the whole set.
