# lumen — Design System

## Design Context

### Users
A solo developer (the owner) using lumen as a daily-driver client over a self-hosted
GitLab instance, triaging issues for defense / space-domain-awareness (SDA) projects.
Usage is focused, repetitive backlog work in an IDE-adjacent context — long sessions,
often a dark editor environment.

### Brand Personality
Precise · calm · operational. A quiet instrument, not a consumer product. It should
feel like trustworthy mission-control software: dense, scannable, fast, with restraint.

### Aesthetic Direction
**Refined dark, Linear-class.** Near-black cool-slate surfaces with subtle elevation;
color carries *meaning* (labels, status, priority) rather than decoration. One rare
amber signal accent. Compact information density. Not the cyan/purple-on-dark AI look.

- **Theme:** dark (chosen for the focused, editor-adjacent viewing context).
- **Density:** compact — maximize scannable rows for backlog triage.

### Type
- **UI / body:** Hanken Grotesk Variable (`--font-sans`) — warm humanist grotesque.
- **Numerals / ids / labels:** Geist Mono Variable (`--font-mono`) — functional tabular
  alignment for issue numbers and counts (not decorative "tech vibes").
- Self-hosted via `@fontsource-variable/*` so they work offline / behind the internal CA.

**Type scale** (`@theme` in `src/styles.css`) — fixed `rem`, never fluid `clamp()`: a dense
instrument UI wants spatial predictability. Every size is a named token; views reference the
token, never an arbitrary `text-[…]`. Two zones with a deliberate gap between them:

| token | size | role |
|-------|------|------|
| `text-micro` | 10px | eyebrow kickers, in-circle glyphs, micro-meta |
| `text-2xs` | 11px | field labels, chips, pills, mono ids |
| `text-xs` | 12px | secondary meta |
| `text-sm` | 14px | dense row body — the workhorse |
| `text-base` | 16px | comfortable detail prose |
| `text-lg` | 18px | section subheads |
| `text-title` | 30px | the page `<h1>` — exactly one per view |
| `text-hero` | 44px | the animated hero count |

`text-title` / `text-hero` / `text-2xs` bake line-height + tracking as Tailwind fallbacks
(`var(--tw-leading,…)`), so a bare `text-title` is correct out of the box but an explicit
`leading-*` still overrides (e.g. the wrapping IssueDetail `<h1>` keeps `leading-[1.08]`).
The page-`<h1>` role is the token's whole reason for being: it used to render 26 / 30 / 32px
across three views — now one token, one size.

### Color (defined in `src/styles.css` under `.dark`)
- OKLCH throughout; neutrals tinted toward **hue 256** (cool slate) for cohesion.
- Background `oklch(0.178 0.008 256)`, card `0.214`, border `0.288`. No pure black/white.
- **Accent:** amber `oklch(0.82 0.142 81)` — used sparingly: the primary/create action,
  focus rings, and the header signal dot. ~10% visual weight at most.
- Semantic label/status/priority colors are brightened for contrast on dark surfaces.

### Design Principles
1. **Color means something.** Reserve hue for state (priority/type/status/labels) and the
   single amber accent. Surfaces stay neutral slate.
2. **Lift signal out of noise.** Parse scoped GitLab labels (`scope::value`); promote
   priority/type/status into glyphs and chips, leave the rest as two-tone pills.
3. **Scannable density.** Tight, consistent rows; mono tabular ids; readable contrast
   (luminance-derived text on every colored chip).
4. **No AI tells.** No side-stripe accents (>1px) on list items, no gradient text, no
   glassmorphism. Indicators are leading glyphs / tints / full borders instead.
5. **Quiet motion.** One staggered list reveal on load (ease-out, opacity+transform,
   reduced-motion safe). No perpetual or bouncy animation — the sole exception is the
   header signal lamp, which breathes *only while a query is in flight* (a loading
   affordance, kin to the spinners), and is steady when idle.

## Delight (subtle, telemetry-flavored — never confetti)
The instrument should feel alive without being noisy. All effects are <300ms, non-blocking,
and reduced-motion safe.
- **Signal lamp = liveness readout** (`App.vue` + `.lamp-busy`): steady amber when idle, a
  slow breath driven by `useIsFetching()` while syncing.
- **Console boot signature** (`main.ts`): a styled amber/mono "lumen ▸ operational" line —
  an easter egg for the dev-owner, discovered not announced.
- **Dynamic tab titles** (`useTitle`): `#iid title · lumen` on detail, `repo · lumen` on a
  list, `Projects · lumen` on the picker. The drawer passes `embedded` so it yields the
  title to the list behind it.
- **Quiet success cues**: Create button flashes `Added ✓` (`IssueList`); comment posts show
  a fading `Posted ✓` (`IssueDetail`) — same idiom as the scratchpad's `Saved`.
- **Hero count rolls** (`Odometer.vue`): the issue/project counts resolve digit by digit —
  each column rolls to its new glyph, rightmost first, and the roll direction tracks the value
  (down when a filter narrows the set). Inherits the host font/size/color; carries an `sr-only`
  plain value; snaps under reduced motion. Supersedes the old `.animate-count` rise+fade (the
  `count-settle` keyframe is retained as an unused, still-valid token).
- **Motion tokens** (`styles.css`): `lamp-breathe`, `count-settle`, `status-in` keyframes →
  `.lamp-busy` / `.animate-count` / `.animate-status` utilities, all behind
  `prefers-reduced-motion: no-preference`.

## Continuity + tactility (the "spice" pass)
Navigation reads as one instrument retuning, not a page swap; the surfaces you touch feel
physical. All View Transitions go through `withViewTransition()` (`src/lib/viewTransition.ts`),
which degrades to a plain, instant update where VT is unavailable or motion is reduced.
- **Row ⇄ board-card morph** (`IssueList.setView`): toggling List ⇄ Board runs inside a View
  Transition; each of the first `VT_CAP` (30) issues carries a shared `view-transition-name`
  (`issue-<iid>`) on both its row (`IssueRow.vtName`) and its board-card wrapper, so the same
  issue FLIPs between layouts. Bounded so a large loaded set never snapshots hundreds of nodes.
- **Tab handoff** (`IssueList`/`PipelineList` `onTabNav`, + the drawer→detail `expandIssue`):
  issues ⇄ pipelines morph the shared `.vt-project-title` and cross-fade the rest — the same
  shared-title idiom as the picker→issues launch. Modified clicks fall through to the real href.
- **Drop settle** (`.animate-drop-in`): a card dropped into a new lane scales-relaxes with a
  fading amber ring (keyed off `justDropped`), so an optimistic retag reads as the card arriving.
- **Filter tokens spring** (`.facet-*` + `TransitionGroup`): active-filter chips scale+fade in
  on add, recoil out on remove, survivors slide to close the gap. Ease-out, no bounce.
- **Default morph cadence** (`::view-transition-group(*)`): 0.34s ease-out for the card FLIP;
  the named `project-title`/`root` groups keep their bespoke timings (more specific selectors).

### Pipelines view (compact rows, telemetry idiom, see `PipelineList.vue`)
- **Compact-by-default rows that expand**: each run is a dense one-line row; the row header
  is the expand toggle (a real `<button>` when there are stages, inert `<div>` otherwise) and
  the full labeled `PipelineStages` stepper drops down via a `grid-rows-[0fr↔1fr]` transition
  (`motion-safe:` so reduced-motion snaps it). Multi-expand (a `Set`), so several can stay open.
- **Stages distilled to a dot strip** (`PipelineStageDots`): the collapsed row shows stages as
  small status dots joined by hairline connectors — the stepper on one line, names/status in
  each dot's tooltip. Shares `TONE_VISUALS` with the full stepper so a dot matches its indicator.
- **The armed bell breathes** (`bell-listen` → `.bell-listening`): subscribing to a running
  pipeline makes its alert bell breathe amber — the *liveness-lamp idiom applied to a
  subscription* ("listening for this run"). The breath onset is the arm confirmation; it
  stops the instant the run leaves flight. The signature moment. Sanctioned sibling of the
  header lamp's documented breathing exception.
- **Watched-run ring**: a subscribed run's row wears a faint `ring-1 ring-primary/20`
  (full ring, never a side-stripe) — a quiet "eye on this" state signal, `transition-shadow`
  so it fades as you arm/disarm.
- **Staggered row entrance**: reuses `.animate-row-in` at the issue-row cadence
  (`index × 26ms`, capped at 14); new rows from a later poll animate in on their own,
  reordered ones don't replay.
- **Status settles on change**: the `PipelineStatusBadge` is keyed on `status` with
  `.animate-status`, so a run advancing (running → passed) replays the settle while
  unchanged polls stay still.
- **Standby empty state**: an idle (steady, not breathing) signal lamp + a `Standby` mono
  eyebrow and operational copy ("…to light this up") instead of a generic "nothing here."
- **`AssigneeAvatar compact`**: avatar-only variant (name on the title tooltip) for dense rows.
- **Two-channel pipeline alerts**: a watched run finishing *always* raises an in-app toast
  (`useToast`/`ToastHost`); an OS notification fires on top *only when the app isn't active*
  (`isAppActive()` — `document.hasFocus()` + visibility), so you're not double-pinged while
  looking at Lumen. The toast carries the status tone + `ref · #iid` and opens the run on click.

## Toasts (`useToast` + `ToastHost`, mounted once in `App.vue`)
A restrained, bottom-right toast stack in the instrument idiom — solid `bg-card` focal cards
(`shadow-pop`), **no glassmorphism**, tone dot/icon (success/failed/info), click-to-open + a
dismiss control, tone-specific auto-dismiss (failures linger at 10s). Module-singleton state
like `useConfirm`; `TransitionGroup` enter/leave behind `motion-safe:`. Reusable beyond
pipelines.

## Reusable pieces
- `src/lib/labels.ts` — scoped-label parsing, `readableText` (WCAG luminance), `darken`,
  `tint`, and `priorityOf` / `typeOf` / `statusOf` / `remainingLabels` semantics.
- `src/lib/issueView.ts` — client-side `sortIssues` (priority/title/updated) and
  `groupIssues` (none/status/priority/assignee), plus the `Facet` type. Sort/group/board
  run on the loaded set because priority & status live in labels (server can't order them).
- `LabelChip.vue` — two-tone scoped pill, contrast-correct, hairline ring.
- `StateBadge.vue` — `compact` dot variant (lists) + full pill (detail).
- `IssueRow.vue` / `IssueCard.vue` — list row & board card. Both use the stretched-link
  pattern (RouterLink overlay + facet buttons above) so clicking a label/priority/status/
  assignee emits a `filter` facet instead of navigating; the row/card still links.

## Issues workspace (not GitLab-shaped)
- **List ⇄ Board** toggle. Board = kanban columns grouped by workflow status
  (on-deck → in-progress → in-review). List = optional grouping + sort menus.
- **Click-to-filter**: any facet on a row/card toggles a filter; active filters show as
  removable tokens. Every facet reduces to a label or assignee filter, so the GraphQL
  query needs no new variables.
- Issues infinite-load (50/page) so client-side sort/group sees the whole set.
